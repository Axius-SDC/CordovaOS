# XSD Generation Audit Report — CordovaOS

## Date: 2026-03-06

Comprehensive audit of all 10 domain XSD schemas, XML instances, and JSON instances generated by SDCStudio, compared against the design templates in `/home/twcook/GitHub/SDC_Demo/templates/`.

## Systematic Bugs (All Domains)

### BUG A — CRITICAL: Sub-clusters never nested inside parent clusters

Every domain with sub-clusters has this problem. Sub-clusters are generated as standalone `ClusterType` definitions in the XSD but are never referenced as children of their parent cluster's `xsd:sequence`.

**Root cause**: The cluster hierarchy builder in the generator emits sub-clusters as top-level XSD types but does not add `<xsd:element ref="ms-{subclusster_cuid}"/>` inside the parent cluster's sequence.

**Impact**: ~50% of all critical findings trace to this single bug.

**Affected domains**: All 10 (Civil Registry, Healthcare, Education, Employment, Business Registry, Law Enforcement, Property Registry, Tax & Revenue, Vital Statistics, Maritime).

### BUG B — CRITICAL: XML/JSON instances only contain root cluster components

Because sub-clusters aren't nested, the XML and JSON instance generators only emit the root cluster's direct atomic components. All sub-cluster content is completely absent.

**Impact**: Instance documents contain 5-60% of expected data depending on domain.

### BUG C — SIGNIFICANT: Person-reference sub-clusters merged into Party structures

When template sub-clusters contain person identifiers (Officer Badge Number, Officer Name, Crew Member Name), the agentic pipeline converts them to Party/Participation structures instead of data sub-clusters.

**Lost components**: Officer Badge/Rank/Unit (Law Enforcement), Person Full Name (Maritime, Business Registry, Vital Stats), Crew List structure (Maritime).

### BUG D — MODERATE: xdtoken-language systematically omitted from XML instances

All XdToken elements show `<!-- Element xdtoken-language not yet implemented -->` in XML instances.

### BUG E — MODERATE: Enumeration drift from template specifications

Reused components (@NIH_CDE, @NIEM) bring their own enumerations that differ from templates. LLM enrichment also changes enumerations.

**Examples**:
- Visit Type: template has routine/urgent/emergency/follow-up/screening; generated has Screening/Baseline/Follow-up/Unscheduled/Early termination/Final
- Cargo Type: template has container/bulk/liquid/break-bulk/vehicles/livestock/hazardous; generated has containers/bulk_dry/bulk_liquid/vehicles/general_cargo/refrigerated/livestock/none
- Payment Status: template has Paid in Full/Partial Payment/Overdue/Payment Plan/Waived; generated has Paid/Invoiced/Overdue

### BUG F — MODERATE: maxLength values loosened

Several components have higher maxLength than template spec. Consistently in one direction (looser).

| Component | Template | Generated |
|-----------|----------|-----------|
| Dosage | 100 | 200 |
| Allergy Description | 300 | 500 |
| Port Call ID | 20 | 50 |
| Flag State | 100 | 200 |
| Lot Number | 50 | 200 |
| Student ID | 20 | 50 |

### BUG G — MODERATE: Spurious component generation

The agentic system invents components not in templates:
- Medication Dosage Amount (XdQuantity) — Healthcare
- Crew Count, Passenger Count (XdCount) — Maritime
- Container Count (XdCount) — Maritime

## Root Cause Priority (for SDCStudio fixes)

1. **Sub-cluster nesting** — Generator must add sub-cluster references inside parent cluster sequences
2. **XML/JSON instance traversal** — Instance generators must recurse into sub-clusters
3. **Party/data confusion** — Template sub-clusters with person references should remain data sub-clusters, not become Party structures
4. **xdtoken-language** — XML instance generator needs xdtoken-language support
5. **Template fidelity** — maxLength, enumerations, component omissions are secondary but real
6. **XdCount generation** — Tax Year and Assessment Year (XdCount type) are never generated; may be a broken code path

## Impact on CordovaOS Demo

Despite these bugs, the **schema-level RDF** (JSON-LD, SHACL) is complete enough for SPARQL queries. The cross-domain join mechanism (shared National ID and Business Registry Number ct_ids, Party identity) works correctly.

For the demo, we have two options:
1. **Fix the generator** and regenerate all 10 apps (thorough, but delays demo)
2. **Manually augment** the generated schemas to add missing sub-cluster nesting (faster for demo, documents bugs for SDCStudio backlog)

Recommendation: Option 2 for the demo deadline, with all bugs filed as SDCStudio issues.
